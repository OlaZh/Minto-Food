# Minto Food + Minto Fit — Shared Health Data Plan

## Мета

Зробити так, щоб `Minto Food` і `Minto Fit` працювали з одними й тими самими health-даними без дублювання, конфліктів і втрати самостійності кожного застосунку.

Потрібно отримати:

- єдине джерело правди для ваги, замірів тіла, базових параметрів і активностей;
- спільний календар активностей за рік;
- окремі ролі для двох застосунків:
  - `Minto Food` — харчування, схуднення, щоденна активність;
  - `Minto Fit` — тренування, програми, зал, домашні плани;
- відсутність дублювання одних і тих самих записів у двох місцях.

## Що вже є

### Minto Food

- `user_profiles` — health/profile data
- `weight_records` — історія ваги
- `user_activities` — ручні активності

### Minto Fit

- `mf_body_stats` — вага + заміри тіла
- `mf_workouts` — тренування
- `mf_workout_sets` — підходи

## Ключовий принцип

Не робити app-to-app передачу даних за схемою:

- `Fit -> Food`
- `Food -> Fit`

Замість цього зробити:

- спільні таблиці як єдине джерело правди;
- обидва застосунки читають і пишуть у ці таблиці;
- якщо дані створені в одному застосунку, другий просто їх бачить.

## Продуктове правило

Активності не прибираємо з `Minto Food`.

Причина:

- є користувачі, яким потрібне лише харчування + кроки + прогулянки + схуднення;
- їм не можна нав’язувати другий застосунок тільки для базової активності.

Отже:

- `Minto Food` лишається самодостатнім для схуднення;
- `Minto Fit` лишається самодостатнім для тренувань;
- обидва працюють на спільному health-core.

## Цільова модель даних

### 1. Shared profile core

Потрібне одне канонічне місце для:

- вік;
- зріст;
- ціль;
- цільова вага;
- можливо стать;
- можливо базові налаштування, які впливають на розрахунки.

Результат:

- обидва застосунки читають одні й ті самі базові параметри;
- розрахунки не роз’їжджаються.

### 2. Shared body metrics

Потрібне одне канонічне місце для:

- ваги;
- замірів тіла;
- історії змін по датах.

Рекомендація:

- використати одну історичну таблицю записів body metrics;
- не зберігати окремо ще одну “поточну вагу” в іншій таблиці без потреби.

### 3. Shared activity timeline

Потрібне одне канонічне місце для всіх активностей:

- зал;
- пілатес;
- йога;
- домашні тренування;
- прогулянки;
- біг;
- велосипед;
- кроки;
- інші ручні активності.

Саме ця модель потрібна для єдиного календаря на рік.

### 4. Fit-specific training domain

У `Minto Fit` окремими залишаються:

- `mf_programs`
- `mf_program_exercises`
- `mf_exercises`
- `mf_workout_sets`

Це внутрішній тренувальний домен `Fit`.

### 5. Food-specific nutrition domain

У `Minto Food` окремими залишаються:

- їжа;
- меню;
- meals;
- вода;
- рецепти;
- статистика харчування.

## Рекомендований розподіл відповідальності

### Minto Food створює

- кроки;
- прогулянки;
- біг;
- велосипед;
- плавання;
- інші щоденні або ручні активності;
- базові profile-дані для користувачів, які взагалі не користуються `Fit`.

### Minto Fit створює

- тренування в залі;
- домашні плани;
- пілатес;
- йогу;
- структуровані workout sessions;
- заміри тіла, якщо користувач вносить їх у `Fit`.

### Обидва застосунки читають

- вагу;
- заміри;
- базові profile-параметри;
- загальний календар активностей;
- сумарні спалені калорії по активностях.

## Покроковий план

## Фаза 1 — Зафіксувати модель

### Крок 1. Підтвердити спільні сутності

Затвердити, що shared між двома застосунками:

- базові параметри профілю;
- вага;
- заміри тіла;
- activity timeline;
- правила розрахунку добової норми.

### Крок 2. Підтвердити, що лишається окремим

Затвердити, що окремими лишаються:

- nutrition domain у `Food`;
- workout/program domain у `Fit`.

### Крок 3. Прийняти правило “single source of truth”

Для кожної категорії визначити єдине джерело правди:

- profile base;
- body metrics;
- activities;
- calorie target logic.

Без цього далі йти не можна.

## Фаза 2 — Спроєктувати shared schema

### Крок 4. Спроєктувати shared profile table або profile layer

Потрібно описати:

- які поля обов’язкові;
- які опціональні;
- які використовує `Food`;
- які використовує `Fit`;
- які поля впливають на калорії й цілі.

### Крок 5. Спроєктувати shared body metrics table

Потрібно вирішити:

- чи мігруємо `weight_records` + `mf_body_stats` у нову спільну таблицю;
- чи адаптуємо одну з існуючих таблиць під shared use.

Рекомендація:

- одна історична таблиця з `recorded_at` і всіма замірами в одному записі.

### Крок 6. Спроєктувати shared activities table

Таблиця має підтримувати:

- `user_id`
- `source_app`
- `source_type`
- `source_id`
- `activity_type`
- `title`
- `started_at`
- `duration_minutes`
- `calories_burned`
- `distance_km` або `steps` за потреби
- `metadata`

Мета:

- з одного місця будувати календар, summary і вплив на добову норму.

### Крок 7. Визначити мапінг workout -> shared activity

Потрібно описати правило:

- кожне завершене тренування `mf_workouts` створює один shared activity event;
- цей event не дублюється вручну в `Food`.

## Фаза 3 — Спроєктувати міграцію без втрати даних

### Крок 8. Аудит поточних таблиць і полів

Потрібно звірити:

- `user_profiles`
- `weight_records`
- `user_activities`
- `mf_body_stats`
- `mf_workouts`

І скласти таблицю:

- що переносимо;
- що залишаємо;
- що архівуємо;
- що стає read-only.

### Крок 9. План переносу ваги і замірів

Потрібно окремо вирішити:

- звідки беремо історію ваги;
- як переносимо заміри з `mf_body_stats`;
- як не створити дублікати за однакові дати.

### Крок 10. План переносу активностей

Потрібно:

- перенести `user_activities` у shared timeline;
- перетворити завершені `mf_workouts` у shared activity entries;
- додати поле походження запису.

### Крок 11. Підготувати rollback strategy

Перед запуском:

- мати SQL rollback;
- мати backup план;
- не видаляти старі таблиці одразу.

## Фаза 4 — Backend / DB implementation

### Крок 12. Створити нові shared таблиці або shared view-layer

Залежно від затвердженої моделі:

- створити таблиці;
- додати індекси;
- додати RLS;
- додати constraints.

### Крок 13. Додати DB-функції або view для календаря

Потрібно мати один стабільний спосіб отримати:

- усі активності за день;
- усі активності за місяць;
- усі активності за рік;
- агрегати для кольорового календаря.

### Крок 14. Винести логіку добової норми з фронтенду

Зараз у `Food` формула сидить у фронтенді.

Потрібно перейти до одного з варіантів:

- серверна функція / RPC;
- єдиний shared calculation module;
- materialized logic у БД або backend layer.

Мета:

- щоб `Food` і `Fit` не рахували по-різному.

## Фаза 5 — Інтеграція Minto Food

### Крок 15. Перевести профіль `Food` на shared profile/body data

`Food` має:

- читати спільні profile-поля;
- читати спільну вагу й заміри;
- більше не тримати дубль там, де він не потрібен.

### Крок 16. Перевести `Food`-активності на shared activity timeline

Усі ручні активності в `Food` мають писатися в shared activity layer.

### Крок 17. Оновити блок впливу активності на денну норму

`Food` має брати активності не лише зі своїх локальних записів, а зі спільної історії.

## Фаза 6 — Інтеграція Minto Fit

### Крок 18. Перевести `Fit`-профіль на shared profile/body data

Зараз частина profile-даних у `Fit` живе в `localStorage`.

Потрібно перевести в shared persistent layer:

- age;
- height;
- goal;
- goal weight.

### Крок 19. Зв’язати `mf_workouts` зі shared activities

При завершенні тренування:

- або створюється shared activity event;
- або `mf_workouts` стає джерелом для derived shared view.

Рекомендація:

- створювати явний shared activity event.

### Крок 20. Перевести body stats `Fit` на shared body metrics

Внесення ваги/замірів у `Fit` має зберігатися в ту ж канонічну таблицю, що й у `Food`.

## Фаза 7 — Єдиний календар активностей

### Крок 21. Визначити типи подій і кольори

Потрібно узгодити:

- зал;
- пілатес;
- йога;
- прогулянка;
- біг;
- велосипед;
- кроки;
- інше.

Для кожного:

- label;
- icon;
- color;
- чи показуємо калорії;
- чи показуємо duration;
- чи показуємо special badge.

### Крок 22. Реалізувати unified year calendar

Календар має показувати:

- наявність активності в день;
- тип активності;
- якщо в день було кілька активностей — або пріоритетний тип, або mixed state;
- tooltip / detail list на день.

### Крок 23. Визначити правила multi-activity day

Окремо вирішити:

- якщо в день були і зал, і прогулянка, який колір показує календар;
- чи показуємо stacked dots;
- чи показуємо “mixed” стан.

## Фаза 8 — Cleanup

### Крок 24. Після стабілізації перевести старі таблиці в legacy status

Після перевірки:

- старі джерела або freeze;
- або використовувати лише для читання;
- або видаляти в окремій міграції.

### Крок 25. Оновити GDPR export / delete

Потрібно додати в export і delete:

- shared body metrics;
- shared activities;
- нові profile-поля.

## Фаза 9 — Тестування

### Крок 26. Перевірити сценарії одного застосунку

Сценарії:

- користувач має лише `Food`;
- користувач має лише `Fit`.

### Крок 27. Перевірити сценарії двох застосунків

Сценарії:

- вага внесена у `Fit`, видно в `Food`;
- прогулянка внесена у `Food`, видно в `Fit`;
- тренування із залу видно в `Food` календарі;
- добова норма не конфліктує.

### Крок 28. Перевірити антидублікати

Треба перевірити:

- чи не дублюється “зал”;
- чи не дублюється вага;
- чи не дублюються заміри за одну дату.

## Ризики

### Ризик 1. Подвійне джерело правди

Якщо лишити старі таблиці активними паралельно, конфлікти повернуться.

### Ризик 2. Дублікати активностей

Особливо для:

- тренувань залу;
- ручного внесення однакових подій;
- імпорту старих даних.

### Ризик 3. Різні формули калорій

Якщо `Food` і `Fit` рахують по-різному, shared data не врятує від хаосу.

### Ризик 4. Неповна міграція профільних полів

Якщо `Fit` і далі триматиме частину важливих даних тільки в `localStorage`, повної синхронності не буде.

## Рішення, які треба окремо затвердити перед реалізацією

1. Яка саме таблиця стане канонічною для ваги і замірів:
   - нова shared таблиця;
   - або еволюція `mf_body_stats`;
   - або еволюція `weight_records` + додаткові поля.

2. Що робимо зі старою `user_activities`:
   - мігруємо в нову shared table;
   - чи тимчасово лишаємо як legacy-source.

3. Де живе формула добової норми:
   - БД / RPC;
   - shared backend logic;
   - shared client module.

4. Як показувати кілька активностей в одному дні в річному календарі.

## Рекомендований порядок старту

Щоб не розмазувати роботу, робимо в такому порядку:

1. Затверджуємо shared model.
2. Проєктуємо shared tables.
3. Переводимо вагу + заміри.
4. Переводимо activities.
5. Робимо unified calendar.
6. Лише потім чистимо legacy.

## Мінімальний перший інкремент

Якщо йти без перевантаження, найкращий перший реліз такий:

1. Зробити shared activity layer.
2. Показати в календарі `Fit` і зал, і прогулянки/вело/біг із `Food`.
3. Не чіпати ще глибоко всю формулу калорій.
4. Після цього окремим етапом перевести вагу й заміри в shared body layer.

Це дасть користь найшвидше:

- одразу буде загальна картина активності;
- менше ризику;
- не треба одним махом ламати весь profile flow.

## Що ще потрібно зробити саме в Minto Fit

### Критично для зв’язки з Minto Food

- додати в `Fit` окремий daily-summary block, але всі значення брати з `Food`;
- у цьому блоці показувати:
  - добову норму калорій;
  - з’їдено за день;
  - спалено за активності;
  - залишок / перевищення;
- не дублювати в `Fit` власну формулу добової норми, якщо canonical-логіка лишається у `Food` або shared layer;
- перевести `Fit`-поля `age`, `height`, `goal`, `goal weight` з `localStorage` у shared profile layer;
- перевести читання `goal weight` у графіках `Fit` зі `localStorage` на shared source;
- визначити єдину точку запису для ваги й замірів, щоб `Fit` не жив окремим body-flow;
- кожне завершене тренування `mf_workouts` має ставати shared activity event з полями походження запису, щоб не було дублювань.

### Критично для єдиного календаря активностей

- розширити календар у `Fit`, щоб він показував не лише `mf_workouts`, а всі активності зі shared activity layer;
- додати підтримку різних типів подій:
  - зал;
  - домашнє тренування;
  - пілатес;
  - йога;
  - прогулянка;
  - біг;
  - велосипед;
  - кроки;
- визначити правила для дня, де є кілька активностей;
- при натисканні на день показувати не одну подію, а повний список активностей за день;
- у деталях дня показувати мінімум:
  - тип активності;
  - назву;
  - тривалість;
  - калорії;
  - джерело запису (`Food` / `Fit`);
- зробити так, щоб зал і не-зальні активності давали одну загальну картину року, а не два окремі світи.

### Важливо для самого продукту Minto Fit

- додати в `Fit` компактний блок “Сьогодні”, який дає коротку health-картину дня без переходу в `Food`;
- показувати в `Fit` не тільки факт тренування, а й тренд по спалених калоріях;
- розширити прогрес не лише до “був/не був у залі”, а до аналітики по типах активності;
- краще зв’язати body progress і training progress, щоб користувач бачив:
  - як змінюється вага;
  - як змінюються заміри;
  - чи є кореляція з частотою тренувань;
- довести до кінця UX для time-based / home-based сценаріїв, щоб `Fit` був однаково зручний не лише для силових програм;
- підготувати нормальну документацію `Fit`, бо зараз технічний onboarding і логіка продукту описані недостатньо.

### Після базової інтеграції

- додати більш глибокі personal records, не лише по вазі, а й по повтореннях / об’єму / тривалості;
- додати більш корисні week/month summaries;
- продумати редагування / видалення / історію body entries;
- продумати, чи потрібен окремий activity screen у `Fit`, чи достатньо календаря + day details + summary blocks.
